## 1. Architecture Overview

### 1.1 System Context

The Multi-Agent Tour Guide System is an enterprise-grade AI orchestration platform that coordinates multiple specialized agents to deliver personalized travel content.

### 1.2 Key Architectural Principles

| Principle | Implementation |
|-----------|---------------|
| **Separation of Concerns** | Each agent handles one content type |
| **Single Responsibility** | Components do one thing well |
| **Open/Closed** | Plugin architecture for extensibility |
| **Dependency Inversion** | DI container for loose coupling |
| **Fail-Safe** | Graceful degradation at every level |

### 3.3 Base Agent

```python
class BaseAgent(ABC):

    def __init__(self, agent_type: str):

        """Search for content - implemented by subclasses"""
        pass
    
    def execute(self, point: RoutePoint) -> Optional[ContentResult]:
        """Execute agent task with logging and error handling"""
        try:
            result = self._search_content(point)
            return result

            logger.error(f"Agent {self.agent_type} failed: {e}")
            return None
```

### 7.3 Retry with Backoff

```python
class RetryPolicy:
    """Exponential backoff retry strategy."""
    
    MAX_RETRIES = 3
    BASE_DELAY = 1.0      # 1 second
    MAX_DELAY = 16.0      # 16 seconds
    EXPONENTIAL_BASE = 2
    
    # Delay sequence: 1s â†’ 2s â†’ 4s
    def get_delay(self, attempt: int) -> float:
        delay = self.BASE_DELAY * (self.EXPONENTIAL_BASE ** attempt)
        return min(delay, self.MAX_DELAY)
```

## 11. Design Decisions (ADRs)

### ADR-001: Queue-Based Synchronization

**Status:** Accepted

**Context:** Need to synchronize multiple agents before Judge evaluation.

**Decision:** Use a Smart Queue with tiered timeouts (15s soft, 30s hard).

**Consequences:**
- âœ… Judge always sees all available results together
- âœ… System never blocks forever
- âœ… Graceful degradation when agents fail
- âš ï¸ Added complexity in queue management

---

### ADR-002: LLM Provider Priority

**Status:** Accepted

**Context:** Need to support multiple LLM providers.

**Decision:** Anthropic Claude is primary, OpenAI GPT is fallback.

**Consequences:**
- âœ… Better reasoning quality with Claude
- âœ… Fallback ensures availability
- âš ï¸ Different prompt tuning needed per provider

---

### ADR-003: Plugin Architecture

**Status:** Accepted

**Context:** Need to extend system without modifying core code.

**Decision:** Auto-discovering plugin system with YAML manifests.

**Consequences:**
- âœ… Open/Closed principle satisfied
- âœ… Third-party extensibility
- âœ… Clean separation of concerns
- âš ï¸ Plugin compatibility management needed

---

### ADR-004: ThreadPoolExecutor vs AsyncIO

**Status:** Accepted

**Context:** Need concurrent agent execution.

**Decision:** Use ThreadPoolExecutor for I/O-bound LLM calls.

**Consequences:**
- âœ… Simple, well-understood model
- âœ… Works with sync libraries
- âš ï¸ Thread overhead for high concurrency
- Future: Consider asyncio for API server
